# Review-Gated Lifecycle Ownership

Phase status transitions are split across three actors so that no single step can silently advance or close work without durable evidence. `b-build` owns only `pending → in-progress`; `b-review` owns the verdict and writes exactly one `review-pass` artifact with an implementation fingerprint; `b-save` owns closeout (`in-progress → completed`) and may only close a unit when a matching active review-pass exists, the fingerprint still matches implementation paths, and no active iterate addresses the target.

`b-save` has two modes: **checkpoint** (interrupted or unreviewed work — memory only, no completion mutation) and **accepted closeout** (valid pass → atomic phase/overview/backlog/memory transition, review-pass completed last as recovery marker). Intermediate phased closeout promotes exactly the first dependency-ready next phase and leaves the subject active; final/non-phased closeout completes parents and the subject only when no other unit remains active. Save reports a staging checklist but never stages — the durable edge is `review → docs if flagged → save → explicit stage → commit`.

Letting `b-build` mark completion was rejected because it claimed success before the mandatory review gate and left no durable evidence that a review occurred. Inferring completion from checkboxes, chat, or changed files alone was rejected for the same reason: only a current review-pass is acceptance evidence.
