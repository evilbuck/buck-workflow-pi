The Problem

Why Chat-Only Agent Work Breaks Down

AI agents are fast — but speed is not the same as continuity, trust, or repeatability. When the conversation ends, the context evaporates.

🕳️

Lost Context

Chat windows compact. Sessions expire. Critical decisions vanish into scrollback. Next session starts from scratch.

👁️

No Audit Trail

"Why did we do this?" — the only answer lives in someone's chat history. No paper trail, no decision record.

🔁

Repeated Discovery

Each new session re-explores the same codebase, re-learns the same constraints, re-makes the same mistakes.

⚡

Premature Coding

Without guardrails, agents jump to editing files before understanding the problem. Research and planning get skipped.

🤝

No Handoff

Work can't be resumed by another engineer or agent. Everything is trapped in one conversation's context window.

📋

No Quality Gates

Review is optional and easily skipped. Feedback lives in chat comments, not as actionable follow-up items.

The principle: Don't lose work.

Buck workflow gives your AI agent a project notebook, a filing system, and quality gates — so every plan, decision, and line of code survives the session.

✦ Not a framework. A toolkit.

Pick the stages your task needs. Skip the rest. Start in the middle. Finish early. The artifacts and global conventions make it work regardless of which stages you run.

Quick fix /b-build → /b-review
Planned work /b-plan → /b-build → /b-review → /b-save
Full feature /b-brainstorm → /b-research → /b-plan → /b-build → /b-review
Ad-hoc Just work. The baseline handles durability.

The Skills

Named Thinking Modes

18 skills across 7 categories. Each is a different kind of thinking with its own write boundary. Pick the ones your task needs. They're pure Markdown — portable across Pi, Claude Code, Cursor, OpenCode, and Codex.

Intake & Discovery
/b-brainstorm
Interview-style capture of vague ideas. One question at a time, then writes a loose first-draft plan.
/b-explore
Trace internal code architecture, map data flows, find entry points. Strictly read-only on the codebase.
/b-research
Investigate external sources — APIs, libraries, docs, web. Captures evidence incrementally.
Planning
/b-plan
Turn context into a bounded plan: scope, affected files, implementation steps, risks, verification.
/b-phase
Decompose a large plan into sequential, independently-verifiable phases with dependencies.
Building
/b-build
Standard implementation with TDD red-green-refactor. Smallest safe change. Follows existing patterns.
/b-build-hard
Complex, ambiguous, or higher-risk work. Thinks through trade-offs, breaks changes into safe steps.
/b-iterate
Quick follow-up fixes, polish, and review-loop edits. Resolves iterate-*.md artifacts in priority order.
Review
/b-review
Validate against a plan or spec. Finds issues, writes iteration artifacts. Quality feedback becomes durable work.
Grilling — Stress-Test Your Plan
/b-grill-me
Interviews you relentlessly, walking every branch of the decision tree. Evaluates if phasing is warranted.
/b-grill-auto
Sends questions to a different AI model via RPC. Second opinion without your context anchoring the answers.
/b-grill-with-docs
Same as grill-me but challenges your plan against existing domain docs. Sharpens vocabulary and boundaries.
Output & Utilities
/b-present
Generate an async-readable presentation package from artifacts. For stakeholders who don't read .context/.
/b-blueprint
Single-page architecture blueprint HTML with Mermaid diagrams and code snippets.
/git-commit
Conventional Commits message from staged changes. Reads draft-commit.md if present.
Benefits

Feature → Benefit → Why You Should Care

Every feature in Buck workflow maps to a concrete benefit that makes AI-assisted engineering more reliable.

01

Continuity

Subject folders with entry points create a stable workspace. Anyone — human or agent — can find the current story without rereading chat.

"Work survives chat compaction, lunch breaks, weekends, and agent handoffs."
02

Trust & Auditability

Memory files with frontmatter create an auditable timeline. Every decision, abandoned approach, and test result is on record.

"Answer 'why did we do this?' without reconstructing from memory."
03

Repeatability

Skills encode reusable guidance. Fix a workflow once, it stays fixed across projects and agents.

"Improve the system once, benefit every future run."
04

Quality Control

Named stages with write boundaries prevent premature coding and skipped review. Findings become artifacts, not vanishing comments.

"Quality feedback becomes part of the workflow state."
05

Lower Rediscovery Cost

Subject entrypoints and memory reduce repeated context gathering. Future sessions start faster because they read what was already learned.

"Each session starts closer to the problem."
06

Handoff & Scaling

b-phase decomposes large plans into safe, resumable chunks. New engineers or agents pick up where the last session ended.

"Quick fix to multi-session — same workflow."

Get Started

Up and Running in 3 Steps

Install the package, add the bootstrap instructions, and start your first workflow.

1

Install the Buck Workflow package

If you're using Pi, install from the GitHub repository:

pi install git:github.com/evilbuck/buck-workflow-pi
2

Install the agent bootstrap instructions

Copy the global AGENTS.md to tell your agent about .context/ conventions:

cp GLOBAL_OR_PROJECT-AGENTS.md ~/.pi/agent/AGENTS.md
3

Start your first workflow

Open any project and run a Buck command. Here's a simple flow for a planned feature:

# Plan the work /b-plan Add user authentication # Start a new session for clean context /new # Build it /b-build .context/2026-06-04.add-user-auth # Review against the plan /b-review .context/2026-06-04.add-user-auth

✦ No workflow is too small

Just fixing a bug? Skip the planning. Run /b-build → /b-review and you're done. The global baseline still ensures memory gets written. You can always add more structure later.
