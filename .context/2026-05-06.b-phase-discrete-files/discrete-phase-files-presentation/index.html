Overview

Give b-phase discrete phase files (one file per phase) and make b-build/b-build-hard update phase state on completion — so any future b-* session can pick up exactly where the last one left off.

The Buck workflow's phasing system now uses separate files per phase instead of embedding all phases in one file. This makes session resume reliable: a cold-start agent can understand exactly where work is without any conversation history.

Why This Matters

Before: Single-File Format
  • All phases embedded in one plan-*-phases.md
  • Acceptance criteria checked inline with [ ] / [x]
  • No durable per-phase state
  • Active phase determined by scanning checkboxes
  • Model auto-switch worked but resume was fragile
  • State only existed in conversation context
After: Discrete Phase Files
  • One phase-N-*.md file per phase
  • Each file has its own frontmatter state
  • Status: pending → in-progress → completed
  • Active phase found by reading summary table
  • Full cold-start resumability
  • State persists across sessions, commits, and context compaction
ℹ️ The old format is still supported. The extension detects which format is in use automatically.

Discrete Phase File Format

Each phase gets its own file with frontmatter tracking and full implementation details.

Frontmatter Fields

--- status: pending # pending | in-progress | completed phase: 1 order: 1 plan: plan-payment-integration.md phases_overview: plan-payment-integration-phases.md difficulty: medium # easy | medium | hard model_hint: capable general model preferred buck_hint: /b-build goal: "Define payment API schema and database migrations" files: [src/api/payment.ts, src/db/migrations/001.sql] from_plan_steps: [3, 4, 5] depends_on: [] # phase numbers; empty = none dependency_type: NONE # HARD | SOFT | NONE acceptance_criteria: - "[ ] PaymentSchema type defined in payment.ts" - "[ ] Migration 001.sql creates payments table" - "[ ] API endpoint POST /payments returns 201" completed_at: null completed_by: null ---

File Body Structure

The body contains the full implementation details for this phase:

  • Context — Why this phase exists, what it builds on
  • Implementation Details — Step-by-step instructions from the original plan
  • Risks — Phase-specific risks and mitigations
  • Verification — How to confirm this phase is complete

Agent Responsibilities

/skill:b-phase
  • Creates one phase-N-*.md per phase
  • Creates overview plan-*-phases.md
  • Adds format: discrete to frontmatter
  • Links phase files in summary table
  • Creates backlog items per phase
/b-build or /b-build-hard
  • Reads overview → finds active phase
  • Marks phase in-progress at start
  • Executes only current phase scope
  • Checks all acceptance criteria on finish
  • Updates status: completed + completed_at
  • Updates overview summary table
/b-save (responsibility 9)
  • Verifies phase file states match reality
  • Upgrades stale in-progress → completed
  • Fixes stale overview summary table
  • Safe net for manual edits and interrupted sessions

Cold-Start Resume

A fresh agent with zero conversation history can pick up exactly where work left off by reading only two files.

Read the phases overview
Agent opens plan-*-phases.md and scans the summary table. The first row without a ✓ Completed badge is the active phase.
Read the active phase file
Agent opens the linked phase-N-*.md file. Frontmatter shows status: in-progress (or pending), difficulty, files, from_plan_steps, and all acceptance criteria. Body contains full implementation details.
Execute
Agent has everything needed: what to do, which files to touch, what "done" looks like. No guessing, no asking the user, no conversation history required.
✅ This works regardless of context window size, session age, or whether the agent has ever interacted with this project before. The files are the source of truth.

Extension Changes

The model auto-switch in extensions/index.ts was updated to support both formats with a dual-path reader.

What Changed

findActivePhaseDifficulty() was replaced with two dedicated functions:

  • findActivePhaseDiscrete() — reads discrete phase files; finds first non-completed phase, returns its difficulty
  • findActivePhaseLegacy() — the original inline checkbox scanner for legacy single-file format

Format detection: if the overview has format: discrete in frontmatter → discrete path. Otherwise → legacy path.

Show dual-path logic pseudocode
function findActivePhaseDifficulty(contextDir):
  overviewPath = findMostRecentPhasesOverview(contextDir)
  overviewContent = read(overviewPath)

  # Format detection
  if "format: discrete" in overviewContent:
    return findActivePhaseDiscrete(overviewPath, overviewContent)
  else:
    return findActivePhaseLegacy(overviewContent)

function findActivePhaseDiscrete(overviewPath, overviewContent):
  overviewDir = dirname(overviewPath)

  # Extract phase file paths from summary table
  # Format: [phase-1-slug.md](phase-1-slug.md)
  phaseFiles = extractLinks(overviewContent, pattern)

  # Fallback: scan directory for phase-N-*.md
  if phaseFiles.empty:
    phaseFiles = readdir(overviewDir).filter("phase-\d+-*.md")

  for phaseFile in sorted(phaseFiles):
    content = read(phaseFile)
    status = extractFrontmatter(content, "status")
    if status != "completed":
      return extractFrontmatter(content, "difficulty")
  return null  # all phases complete

Phase File Link Extraction

Phase files are discovered via the summary table link syntax:

| 1: Schema | completed | medium | [phase-1-schema.md](phase-1-schema.md) |

If no linked files found in table, the code falls back to scanning the directory for any file matching phase-\d+-*.md. Both approaches are supported simultaneously.

b-save Phase State Consolidation

b-save gained a 9th responsibility: ensuring phase state is consistent across the overview and individual phase files.

Read all phase files
Scan the subject folder for phase-N-*.md files. For each, read frontmatter to check status.
Verify state against criteria
If a phase shows status: in-progress but all acceptance criteria are checked ([x]), promote it to status: completed and set completed_at: today.
Sync overview table
Read the overview's summary table. For each phase, verify the status column matches the phase file's actual frontmatter state. Update any stale entries.
Skip for legacy format
If the overview has no format: discrete frontmatter (legacy single-file), skip all consolidation. The old format has no discrete phase files to sync.
⚠️ This is the safety net for interrupted sessions. If an agent marks a phase complete but b-save never runs, the next session's b-save will catch it.

Risks & Trade-offs

  • ⚠️
    Migration path — Existing projects with single-file phased plans won't automatically migrate. The extension fallback keeps them working, but users should re-run /skill:b-phase to regenerate in the new format.
  • ⚠️
    State drift — Manual edits to phase files (without going through b-build) can drift from reality. b-save's 9th responsibility is the safety net but requires running b-save after manual work.
  • ⚠️
    File proliferation — A 6-phase plan creates 7 new files (6 phases + 1 overview). Acceptable for the resumability benefit, but the overview file is critical to keep things scannable.
  • ✓
    Backwards compatibility — Legacy single-file format is fully supported. The extension detects format: discrete in frontmatter to choose the read path. No breaking changes.
