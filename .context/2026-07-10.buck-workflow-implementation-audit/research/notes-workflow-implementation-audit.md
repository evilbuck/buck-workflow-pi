# Rolling Notes — Buck Workflow Implementation Audit

## Scope and initial inventory

- User requested a whole-workflow analysis, specifically reviewing documentation and prompt surfaces for improvement opportunities, gaps, and inconsistencies.
- Primary implementation layers identified from repository guidance: canonical workflow logic in `skills/`, thin agent-native wrappers in `prompts/` and `commands/`, and runtime-specific automation in `extensions/`.
- High-value documentation surfaces identified: `docs/buck-workflow.md`, harness-specific docs, context/eval-kernel docs, and the deprecated `docs/b-flow.md` history.
- Initial inventory shows prompt/command mirrors for the main lifecycle, but `skills/b-loop/SKILL.md` currently has no `prompts/b-loop.md` or `commands/b-loop.md`; prior project memory records this as an explicit deferral, not an accidental omission.

Questions to resolve:
1. Do docs, prompts, and canonical skills describe the same state machine and closeout order?
2. Are wrappers consistently thin, and are all advertised entrypoints installable across targets?
3. Are artifact/status/subject contracts coherent end to end?
4. Do autonomous-loop recommendations and actual execution contracts align?
5. Which gaps are design choices, tracked debt, or undocumented divergence?

### Surface map and first divergence signals

Files mapped: `docs/buck-workflow.md`, `docs/context-artifacts.md`, core lifecycle `SKILL.md` files, `prompts/`, `commands/`, `package.json`, and `scripts/install.mjs`.

- `package.json` registers `prompts/` for Pi, `commands/` for OMP, and the whole `skills/` tree for both; this makes prompt/command parity a real runtime contract rather than editorial duplication (`package.json:13-33`).
- `commands/` contains zero-byte entries for most lifecycle commands, consistent with symlinks, but `b-pr.md` and `b-pr-review-2-issues.md` are non-zero custom wrappers and therefore need separate drift review.
- `docs/buck-workflow.md` is large (over 1,300 lines) and carries the state machine, runtime mapping, OMP modes, component details, artifact model, extension ownership, recommended flows, and discoverability in one file. Its version footer still says `Last updated: 2026-06-07`, while `b-loop` was added on 2026-07-05 and intentionally omitted from prompt/command/doc mirrors.
- The documented quick-fix flow currently appears as `/b-iterate → /b-review`, while the broader completion contract ends at `/b-docs` (conditional) → `/b-save` → `/b-commit`; whether the short flow is a loop fragment or a complete workflow is not stated in the heading.
- `b-phase` still embeds shell discovery examples using `ls -lt ... | head`, while current harness policy requires native file/glob tools. This is a portability/maintenance smell in canonical skill instructions, pending cross-harness verification.

### Primary workflow document and core prompt review

Files examined: `docs/buck-workflow.md`, `docs/context-artifacts.md`, and the core `b-plan`, `b-phase`, `b-build`, `b-build-hard`, `b-review`, `b-iterate`, `b-docs`, `b-save`, and `b-commit` prompt wrappers.

- **The two main diagrams do not reach the documented final step.** The Complete Flow ends at `/b-save` → “Done” (`docs/buck-workflow.md:236-241`) and Command-Only Flow ends at Save (`:298-309`), while the same document defines `/b-commit` as the final step and one-commit-per-completed-unit invariant (`:1035-1063`).
- **The Complete Flow has Mermaid node-ID collisions that change its meaning.** `G2` names both `/b-research` and `/skill:b-grill-me` (`:185`, `:194`); `N` names both Implementation and `/b-review` (`:223`, `:227`); `O` names both HTML Presentation and the issues decision (`:225`, `:228`). The resulting rendered graph cannot faithfully express the intended lifecycle.
- **Presentation is routed into implementation review without an implementation.** `M` (`/b-present`) points to both HTML Presentation and `N` (`/b-review`) (`:216`, `:225`, `:227`), but the documented review stage reviews implementation changes, not plans (`:923-925`).
- **Review write behavior contradicts itself.** The detailed docs call `b-review` read-only (`:905`) and then say it writes `iterate-<subject>.md` when issues exist (`:942-946`). The skill has the more precise boundary: read-only unless writing an iteration artifact.
- **Several working slash commands are documented as skill-only.** `prompts/b-phase.md`, `prompts/b-grill-me.md`, and `prompts/b-grill-with-docs.md` exist and are mirrored in `commands/`, but the matrix, quick-reference table, and discoverability section advertise only `/skill:*` forms (`:327-329`, `:376-380`, `:1309-1312`). This conflicts with the repository's stated `/b-` discoverability model.
- **`b-save` breaks the canonical-skill/thin-wrapper architecture.** `prompts/b-save.md` contains the full 11-step behavior rather than loading `skills/b-save/SKILL.md`; the skill in turn describes the prompt as the executed body. Every other reviewed lifecycle prompt is a thin wrapper. This duplicates policy across two nominally authoritative files and creates a predictable drift point.
- **Protected-branch documentation is already drifting.** `skills/git-commit/SKILL.md` guards `main`, `master`, `dev`, and `develop`; `prompts/b-commit.md` and `docs/buck-workflow.md:1065` omit `dev` from their user-facing summaries.
- **The OMP orchestrate example omits the phase commit invariant.** Its abbreviated per-phase sequence stops at `/b-save` (`docs/buck-workflow.md:138-141`) even though the same document requires `/b-commit` for each completed phase/body unit (`:1051-1060`).
- **Artifact schema coverage is partial.** `docs/context-artifacts.md` defines frontmatter only for memory, subject index, research, plan, and backlog items (`:53-127`); the documented folder model also uses specs, phase files, iterate artifacts, brainstorm artifacts, and phase overview files (`docs/buck-workflow.md:1074-1088`). Their status/link contracts are therefore not centralized in the purported current contract reference.
- **The Quick Fix Loop is ambiguous rather than necessarily wrong.** `/b-iterate → /b-review` (`docs/buck-workflow.md:1271-1275`) is valid as a loop fragment, but unlike the adjacent Review Fix Loop it does not show the pass path into conditional docs/save/commit. Rename it as a fragment or complete the flow.

### Lifecycle and packaging contract review

Files examined: current plan/phase/iterate/save/commit skills, README workflow claims, prompt wrappers for phase/grill/explore, and the multi-harness installer registry.

- **The README contradicts the canonical completion contract.** It says a small change can finish at `/b-build → /b-review` (`README.md:9-17`) and repeats that as a quick-fix flow (`:214-224`), while canonical skill/docs flows require conditional docs → save → commit after accepted work. This is more than “flexibility”: stopping there bypasses the repository's defining durability and final-commit guarantees.
- **One README flow asks the wrong component to do plan review.** `/b-plan → /b-review → ...` is labeled “Plan and review without exploration” (`README.md:224`), but `b-review` explicitly reviews implementation and the detailed docs say it does not review plans (`docs/buck-workflow.md:923-925`).
- **Phase/grill `/b-*` wrappers are real and thin.** `prompts/b-phase.md`, `prompts/b-grill-me.md`, and `prompts/b-grill-with-docs.md` load their canonical skills, confirming the primary docs' skill-only presentation is stale rather than a packaging limitation.
- **`b-save` has no plan/subject completion transition.** Its 11 responsibilities update specs, phase files, iterate files, memory, backlog, and indexes, but never set completed plan frontmatter or the subject `index.md` to `completed` (`prompts/b-save.md:6-42`; `skills/b-save/SKILL.md:25-38`). The shared subject protocol says `b-save` owns subject completion, so active subjects can remain active indefinitely.
- **Memory lifecycle is underspecified.** `b-save` always shows newly written session memory as `status: active` (`prompts/b-save.md:10-22`), but the artifact contract permits `active | completed | superseded` and no responsibility explains when a finished session memory becomes `completed`.
- **Phased backlog promotion has no owner.** `b-phase` creates per-phase items but exposes only Phase 1 in `todo.md`; later phases stay commented/upcoming (`skills/b-phase/SKILL.md:373-384`). It later says to queue the next phase (`:426-432`), but neither `b-phase` nor `b-save` specifies the promotion transition after a phase commits.
- **The commit boundary lacks a staging transition.** `/b-commit` refuses to stage and operates only on already staged changes (`skills/git-commit/SKILL.md:12-23`), but every canonical workflow goes directly from `/b-save` to `/b-commit`. Since `/b-save` itself writes new `.context/**` files, the documented flow needs an explicit user-owned staging gate after save; otherwise commit fails or omits the durable artifacts.
- **Commit draft selection bypasses subject resolution.** `git-commit` picks the most recently modified subject folder with `ls -dt ... | head` (`skills/git-commit/SKILL.md:27-38`) instead of using explicit context/current-session/shared subject resolution. Parallel or unrelated context writes can select the wrong `draft-commit.md`.
- **A stale draft can override current staged reality.** If a draft exists, `git-commit` skips diff analysis entirely (`:35-38`) even though the skill's stated input contract is staged changes. Review, docs, or save may have changed scope after the draft was written.
- **Cross-harness installation is knowingly incomplete.** The installer wires full prompt/skill trees for Claude and OpenCode but only the bootstrap for Codex, Pi, and OMP (`scripts/install.mjs:39-89`; Pi/OMP rely on separate package install). README's “all major harnesses” claim therefore overstates Codex behavior: Codex receives workflow names but not the canonical skills. This aligns with the active high-priority multi-harness installer backlog item.
- **`b-plan` repeats a known-bad OMP detector.** Its recommendation table allows the package's `package.json` `omp` field to prove an OMP runtime (`skills/b-plan/SKILL.md:212-229`). The package always declares that metadata, even under Pi/other harnesses; `b-loop` already corrected this exact bug by requiring runtime state. This can generate OMP-only artifacts for non-OMP sessions.

### Build, phase-state, and current OMP runtime review

Files examined: full `b-build` skill, phase-state sections, OMP prompt stubs, `b-loop`, repository eval-kernel docs, and current OMP 16.3.15 eval documentation (`omp://tools/eval.md`).

- **Phase completion advances before review.** `b-build` marks the active phase and overview row `completed` immediately after implementation (`skills/b-build/SKILL.md:273-285`). A no-argument `b-review` resolves a phases overview to the first non-completed phase (`skills/b-review/SKILL.md:26-33`, `:72-79`), so the canonical closeout recommendation can review Phase N+1 instead of the just-built Phase N. Generated phase instructions avoid this only when the caller remembers to pass the phase file explicitly.
- **There is no durable clean-review record.** `b-review` writes an artifact only for in-plan failures; clean passes exist only in chat. Consequently `b-save` cannot establish from artifacts that a phase's required review gate passed. The existing active backlog item for a plan implementation ledger is directly relevant.
- **The current three-state phase model conflates “implemented” and “reviewed.”** With only `pending → in-progress → completed`, builder-owned completion makes recovery advance too early. The boring fix is to leave a phase `in-progress` until review passes and persist review evidence before save/commit, rather than adding status aliases without a reader migration.
- **`b-build` is not actually portable.** The canonical skill hardcodes Vitest, Playwright, npm scripts, localhost:3000, tmux, and this repository's test layout (`skills/b-build/SKILL.md:13-37`, `:60-93`, `:121-219`). It is installed as a generic skill for arbitrary projects. Test framework/server commands should be discovered from the target repo; only behavioral/TDD invariants belong in the portable core.
- **Its UI detector is over-broad.** Any touched JavaScript or TypeScript file is classified as UI and mandates a Playwright-first test (`:28-37`), which incorrectly catches Node services, CLIs, build tooling, and non-rendering libraries.
- **Mandatory approval blocks autonomous execution.** The TDD checklist always requires confirming interface/test choices and getting user approval before code (`:60-68`), even when an approved plan/phase is the explicit input. That conflicts with `orchestrate`/goal no-yield execution. Approval should be conditional on a material unresolved tradeoff.
- **The skill claims a session plugin that the package no longer has.** `b-build` and `b-iterate` say the plugin tracks the session automatically and “handles the rest,” then read/write `current-session.json`; the current runtime docs explicitly say session-state injection is not part of the wired extension (`docs/buck-workflow.md:1205-1211`).
- **Current OMP has broken the checked-in eval-cell contract.** Repo docs were verified against OMP 15.10.0 (`docs/eval-kernel.md:1-5`) and use `llm(...)`, `agent_type=`, and `from prelude import ...`. Installed OMP is 16.3.15; its current eval contract exposes `completion(...)`, `agent=`, injected persistent helpers, and a different tool input/runtime model. Therefore `b-plan`'s generated workflow cell (`skills/b-plan/SKILL.md:256-393`) and `prompts/omp-workflow.md:19-36` are stale and can fail or no-op. This is a behavioral break, not merely stale prose, because `b-plan` emits executable artifacts.
- **The OMP stubs claim activation while documenting non-activation.** `/omp-orchestrate` and `/omp-workflow` begin “You are now in ... mode” (`prompts/omp-orchestrate.md:10-16`; `prompts/omp-workflow.md:11-17`), while the workflow docs say these stubs are observation-only and the user must separately type the magic keyword. `/omp-goal` correctly calls itself a documentation stub.
- **The orchestrate stub is target-project specific and internally tense.** It hardcodes parent verification to `bun check`, `bun test`, and LSP (`prompts/omp-orchestrate.md:18-34`) and simultaneously requires verification after every phase while saying parent checks run once at the end. Commands and gate cadence must be project-discovered and unambiguous.
- **`b-plan` and `b-loop` can recommend different modes.** `b-plan` declares `goal > workflow > orchestrate` but also says first matching rule wins; `b-loop` says “strongest” without defining an ordering and lists orchestrate/workflow/goal in a different order (`skills/b-plan/SKILL.md:219-229`; `skills/b-loop/SKILL.md:198-211`). One shared recommendation table should own precedence and harness detection.

### Artifact validator, loading docs, and supporting-entrypoint review

Files examined: `scripts/context-artifacts.mjs` and tests, `docs/extension-loading.md`, harness docs, all prompt shapes, command file modes, `b-auto-fix`, `b-grill-auto`, `b-blueprint`, `b-issue-create`, and PR-review wrappers.

- **Artifact validation covers only five kinds.** The schema registry/classifier recognizes memory, subject index, research, plan, and backlog item (`scripts/context-artifacts.mjs:66-100`, `:172-186`). Discrete phases, iterate artifacts, specs, grill sessions, brainstorms, issue artifacts, and draft commits are invisible.
- **Phase overview files are misclassified, not skipped.** `plan-*-phases.md` matches the generic `plan-.*` classifier (`:181-183`) and is validated against plan-required fields (`research`, `memory`, etc.) that its own `b-phase` template does not include. This produces structurally inevitable warnings while still failing to validate phase-specific fields/table consistency.
- **Generated artifact indexes are registries, not relational validation.** `generateIndexes` copies frontmatter into JSON but does not validate paths, backlinks, phase count/table mirrors, or `iterations`/`spec` links (`:273-350`). The docs' “hybrid query view” is accurate; treating it as integrity enforcement is not.
- **The validator's severity model is string-coupled.** It decides warning vs error by checking whether an error message contains `missing required field` (`:391-425`), making future schema evolution brittle.
- **Loading documentation is stale in measurable ways.** `docs/extension-loading.md` still says 19 skills / 14 prompts / 14 symlinks (`:105-138`) while the current inventory is much larger; it also says the `omp` manifest lists only extensions, but `package.json:24-33` explicitly lists extensions, commands, and skills. The Pi filter example omits newer core commands. The truth table should be generated or tested, not hand-counted.
- **The command mirror is healthy but has two intentional regular-file exceptions.** `git ls-files -s` and `stat` confirm 20 symlinks plus regular `commands/b-pr.md` and `commands/b-pr-review-2-issues.md`. Current docs claim all command files are symlinks and do not explain these portable-loader exceptions.
- **Prompt architecture has three categories, not one.** Most prompts are thin `$ARGUMENTS` + skill loaders. `b-save`, `b-pr`, and `b-pr-review-2-issues` duplicate operational policy. The three `omp-*` prompts are runtime-contract documentation stubs. Documentation currently presents all as one thin-wrapper model.
- **`b-pr-review-2-issues` has live wrapper/skill drift.** The prompt preserves the intentional rule that a `.context/**` comment reporting leaked secrets remains actionable (`prompts/b-pr-review-2-issues.md:19-23`); the canonical skill says every `.context/**` comment becomes `context_skip` with no exception (`skills/b-pr-review-2-issues/SKILL.md:56-90`).
- **`b-auto-fix` is named and documented as an executor, but the CLI only stages work.** `main()` writes `stages.json` and `completion-plan.json`, marks setup complete, and exits (`skills/b-auto-fix/scripts/auto-fix.ts:398-477`). It never spawns an agent, runs research/plan/build/review, creates a commit, or invokes the exported success/failure routes. `successPath()` can push/create a draft PR only if some external caller has already executed and committed the work (`:492-559`). This is an unfinished orchestration contract, not a missing optional closeout.
- **`b-grill-auto` is an advertised dead surface.** Its skill explicitly delegates runtime behavior to `/b-grill-auto` (`skills/b-grill-auto/SKILL.md:6-10`), but that extension is unwired and there is no prompt/command mirror. A checked-in `skills/b-grill-auto/grill.py` helper does exist, contrary to the original file-presence inference, but `python3 -m py_compile` fails at line 144 because literal patch-marker `+` lines remain. The skill is therefore still not self-sufficient or runnable as advertised.
- **`b-blueprint` advertises `/b-blueprint` without wiring it.** Frontmatter and body promise `/b-blueprint` or `/skill:b-blueprint` (`skills/b-blueprint/SKILL.md:1-17`), but only the skill form exists; no prompt/command mirror is present. Unlike `b-loop`, there is no explicit deferral.
- **Skill-only can be legitimate when named honestly.** `b-issue-create` has no `/b-*` claim in its frontmatter and remains reachable by skill name; its absence from the core slash-command table is a discoverability choice, not proof of broken behavior. `b-loop` likewise documents its skill-only deferral explicitly.

### Bootstrap and wired-extension review

Files examined: installable `GLOBAL_OR_PROJECT-AGENTS.md`, current `extensions/index.ts`, extension tests, and `skills/cross-platform-pi-omp-loading/SKILL.md`.

- **The bootstrap's “universal” status rule is self-contradictory and incompatible with phases.** It says every artifact uses only `draft | active | completed` and explicitly forbids `in-progress` (`GLOBAL_OR_PROJECT-AGENTS.md:55-59`), then immediately allows memory `superseded` (`:63-74`), backlog only `active | completed`, and the canonical phase skill requires `pending | in-progress | completed`. Status vocabularies must be scoped per artifact kind.
- **The bootstrap and `b-save` disagree on who closes plans.** Global before/after rules and quality gate require completed plan/spec status updates (`:16-22`, `:175-185`); the `b-save` prompt updates specs but omits plan and subject-index completion. Behavior currently depends on the global bootstrap compensating for a missing canonical-skill step.
- **The bootstrap “Steps” line is not a state machine.** It lists `/b-build` then `/b-build-hard` sequentially and includes conditional/optional stages as though all run (`:132-139`). This should be a component list or branch-aware flow; otherwise it contradicts the detailed lifecycle.
- **Model auto-switch resolves a different subject than the skills.** The extension scans every dated subject, chooses the lexicographically newest phases overview, and never checks active session/subject/index status (`extensions/index.ts:105-150`). A `/b-review <older-phase-path>` or multiple active subjects can switch models using an unrelated plan.
- **Model selection has weak test coverage at the resolution boundary.** Existing wired-extension tests check that no model is switched when no phase exists, but no test covers multiple subjects, explicit phase targets, completed newest subjects, or numeric phase ordering.
- **Cross-platform authoring guidance describes removed runtime code as current.** `skills/cross-platform-pi-omp-loading/SKILL.md:100-106` says `extensions/index.ts` still registers/dispatches `b-save`, injects `{{SESSION_STATE}}`, and retains `b-mode`/`b-restrict`; current `extensions/index.ts` has none of those, and current docs explicitly classify them removed/unwired. This helper skill can actively teach package authors the wrong architecture.
- **Goal-mode detection also points at unwritten state.** `b-review` checks `current-session.json.goal` (`skills/b-review/SKILL.md:123-138`), but the wired extension deliberately does not create session state and OMP's goal is runtime state. The reliable signals are the active goal tool/system injection or explicit invocation context, not a legacy JSON field.
- Documentation link diagnostics are clean across `docs/**/*.md`, `README.md`, and `GLOBAL_OR_PROJECT-AGENTS.md`; the problems are semantic drift and executable-contract drift, not broken Markdown links.
