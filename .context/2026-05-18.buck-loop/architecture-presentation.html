1. Goal

Make /b-flow run --autonomous execute the current queue through a deterministic per-phase lifecycle:

select phase → boundary check → build → review → iterate as needed → re-review → save → next phase

The loop is artifact-driven, recoverable, cancellable, and safe by default. It reuses the existing b-flow XState architecture rather than introducing a competing orchestration layer.

2. Current Architecture

Parent Machine — createBuckMachine

The parent owns the coarse workflow. It scans context, routes transitions, and invokes the child chunk-queue actor during executingChunks.

┌─────────────────────────────────────────────────────────────────┐ │ createBuckMachine (parent) │ │ │ │ idle ──START──→ recovering ──→ planning ──→ decomposing ──→ executingChunks ─┤ │ │ │ executingChunks invokes child actor: │ │ ┌──────────────────────────────────────────┐ │ │ │ chunk-queue-machine (child) │ │ │ │ │ │ │ │ idle → buildingQueue → selectingNext │ │ │ │ → spawningWorker → awaitingWorker │ │ │ │ → readingResult → verifyingChunk │ │ │ │ → completedChunk → selectingNext ... │ │ │ └──────────────────────────────────────────┘ │ │ │ │ executingChunks ──→ reviewing ──→ saving ──→ done │ │ │ │ │ └──requires replan──→ planning │ └─────────────────────────────────────────────────────────────────┘

Key Limitation

The current chunk-queue-machine treats every queue item identically: spawn → read result → verify → next. It has no concept of:

  • Per-phase build → review → iterate → save lifecycle
  • Worker mode awareness (build vs review vs iterate vs save)
  • Iteration counting or stagnation detection
  • Phase-boundary git safety
  • Active iterate artifact awareness

Layered Ownership Model

4. Lifecycle State Machine

The core of the change: chunk-queue-machine.ts evolves from a generic queue processor into a deterministic per-phase lifecycle actor.

Current vs Proposed States

Current (generic queue)

idle → buildingQueue → selectingNext → spawningWorker → readingResult → verifyingChunk → completedChunk → selectingNext ... → queueExhausted

Proposed (per-phase lifecycle)

selectingNext → checkingPhaseBoundarySafety → buildingPhase → reviewingPhase → iteratingPhase (loop back) → savingPhase → phaseComplete → selectingNext ... → queueExhausted blockedPhase (any guardrail fires)

Deterministic Routing Rules

// Routing decisions inside the lifecycle actor

switch (result) {
  // Build outcomes
  build_success:        → reviewingPhase;

  // Review outcomes
  review_passed:        → savingPhase;
  review_issues:        → iteratingPhase;    // if active iterate exists
  review_requires_replan: → blockedPhase;

  // Iterate outcomes
  iterate_success:      → reviewingPhase;    // re-review after fix

  // Save outcomes
  save_success:         → mark phase completed → selectingNext;

  // Guardrails (checked at every transition)
  max_iterations(5):    → blockedPhase;
  stagnation_detected:  → blockedPhase;
  git_safety_violation: → blockedPhase;
}

Iteration Loop Detail

┌──────────────────────────────┐ │ │ ▼ │ reviewingPhase ──passed──→ savingPhase ──done──→ phaseComplete │ │ │ issues │ ▼ │ check active iterate │ │ │ ├─ found → iteratingPhase ──success──→ reviewingPhase (above) │ │ │ └─iteration++ → persist to orchestration.json │ └─ none/multiple → blockedPhase (ambiguity) Counter: n / 5 max iterations per phase Stagnation: same fingerprint ×3, no changed files ×2, same failure ×3

Parse Logic (verify-result.ts or adjacent module)

function parseReviewResult(raw: string): ReviewOutcome {
  const fm = parseFrontmatter(raw);

  if (fm.requires_replan === true)
    return { type: "REVIEW_REPLAN" };

  if (fm.review_passed === true)
    return { type: "REVIEW_PASSED" };

  if (fm.issues_found && fm.iterate_file)
    return { type: "REVIEW_ISSUES", iterateFile: fm.iterate_file, fingerprint: fm.issue_fingerprint };

  if (fm.review_passed === undefined || fm.mode === undefined)
    return { type: "REVIEW_MALFORMED" };  // block, don't guess

  return { type: "REVIEW_MALFORMED" };
}

The worker layer gains mode awareness. Each mode loads a specific Buck skill and writes mode-specific result frontmatter.

8. Guardrails & Recovery

Guardrail Matrix

GuardrailThresholdChecked AtOutcome
Max iterations 5 per phase (default) review → iterate transition BLOCK
Issue stagnation Same fingerprint ×3 iterate → review transition BLOCK
No-change stagnation No changed files ×2 iterates iterate → review transition BLOCK
Repeated failure Same block reason ×3 Any blocked transition BLOCK
Active iterate not advancing Status unchanged after completed iterate worker iterate → review transition BLOCK
Multiple active iterates >1 status: active for same phase review → iterate transition BLOCK
Phase-boundary git safety Unattributed source changes before new phase phaseComplete → selectingNext transition BLOCK

Recovery Priority

On recovery, reconcile truth sources in priority order: 1. Phase file frontmatter — canonical for completed/not-completed 2. Worker result files — what actually happened 3. Worker audit files — process metadata (pid, timing) 4. orchestration.json active — runtime projection (correctable) Conflicts: ├─ Orphaned audit (no result) → BLOCK with reason ├─ Result disagrees with phase fm → Phase frontmatter wins └─ Projection stale vs artifacts → Correct from artifacts or BLOCK

STOP / PAUSE Behavior

STOP

  • Abort parent machine
  • Kill active worker subprocess (if pid available)
  • Record reconciliation state in audit
  • Clear actor from memory

PAUSE

  • Do not start new workers
  • Finish current worker if active
  • Then hold in current lifecycle state
  • Resume with /b-flow resume
