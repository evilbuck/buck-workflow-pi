Convert Buck workflow prompts into portable skills

2026-05-12 · Subject: prompt-to-skill-portability

active prompts skills portability agents commands plugins

Summary

Buck workflow currently ships as Pi prompt templates in prompts/. This limits portability to Claude Code, OpenCode, and Codex, which each use different extension mechanisms (skills, commands, plugins).

The plan migrates the canonical workflow logic from prompts into portable skills while keeping each agent's native command/prompt surface as thin wrappers. Runtime automation (e.g. /b-save) stays in extensions/plugins.

Why

  • Agent lock-in: Prompt templates are Pi-specific. Claude Code uses commands/, OpenCode uses commands/, Codex uses plugins. None of these agents natively understand Pi prompt templates.
  • Duplicated logic: If we want Buck workflow on multiple agents, we'd have to copy-paste the full instruction set into each agent's format.
  • Skills are portable: The Agent Skills standard is designed for progressive disclosure and works across Pi, Claude Code, and Codex. OpenCode can consume skill-like markdown via commands.

What Changes

Canonical skill layer (new)

Create one portable skill per major workflow:

  • skills/b-brainstorm/SKILL.md
  • skills/b-research/SKILL.md
  • skills/b-plan/SKILL.md
  • skills/b-build/SKILL.md
  • skills/b-build-hard/SKILL.md
  • skills/b-iterate/SKILL.md
  • skills/b-review/SKILL.md
  • Reconcile existing skills/b-present/SKILL.md with prompts/b-present.md

Command / prompt shim layer (thin wrappers)

Keep agent-native commands but shrink them to skill activators:

AgentWrapper TypeExample
Pi Prompt template /b-plan → "Load and follow /skill:b-plan"
Claude Code Command or plugin skill /b-plan or /buck:b-plan → skill activation
OpenCode Command markdown /b-plan → routes to skill instructions
Codex Plugin skill Bundled skill in plugin; slash commands for UI

Extension / plugin layer (unchanged scope)

Runtime automation stays outside skills:

  • /b-save — session persistence, memory updates
  • Background/flow orchestration
  • Tmux/dev-server integration

How It Works

The architecture is a three-layer stack:

Layer 1: Canonical Skills

Self-contained SKILL.md files with frontmatter, role instructions, behavior rules, and output contracts. Agent-neutral terminology where possible.

Layer 2: Native Command Wrappers

Thin prompts/commands per agent that simply invoke or strongly reference the matching skill. The wrapper contains minimal logic — just enough for the agent's command discovery system.

Layer 3: Runtime Extensions/Plugins

TypeScript/JavaScript modules with event hooks, custom tools, persistence, and session automation. This is where /b-save lives.

A typical flow looks like:

  1. User types /b-plan in their agent
  2. Agent resolves the native command/prompt
  3. The wrapper loads the canonical skills/b-plan/SKILL.md
  4. The skill's instructions drive behavior
  5. After implementation, /b-save (extension/plugin) persists state

Risks & Conflicts

Over-conversion risk: Skills are not a replacement for commands/plugins. Converting runtime automation like /b-save into a skill would lose event hooks and persistence. Keep runtime behavior in extensions/plugins.
RiskMitigation
Invocation mismatch across agents Thin wrappers per agent; canonical skill uses neutral terminology
Context bloat from large skills Progressive disclosure: only name/description in system prompt; full skill loads on-demand
Duplicate source of truth Prompts become thin wrappers, not parallel full implementations
Behavior regressions from Pi-specific assumptions Validate old prompt vs new skill for representative workflows
Scope creep into full marketplace packaging Out of scope: full multi-agent marketplace distribution in this pass
