The Problem

Why Chat-Only Agent Work Breaks Down

AI agents are fast — but speed is not the same as continuity, trust, or repeatability. When the conversation ends, the context evaporates.

🕳️

Lost Context

Chat windows compact. Sessions expire. Critical decisions vanish into scrollback. Next session starts from scratch.

👁️

No Audit Trail

"Why did we do this?" — the only answer lives in someone's chat history. No paper trail, no decision record.

🔁

Repeated Discovery

Each new session re-explores the same codebase, re-learns the same constraints, re-makes the same mistakes.

⚡

Premature Coding

Without guardrails, agents jump to editing files before understanding the problem. Research, planning, review get skipped.

🤝

No Handoff

Work can't be resumed by another engineer or agent. Everything is trapped in one conversation's context window.

📋

No Quality Gates

Review is optional and easily skipped. Feedback lives in chat comments, not as actionable follow-up items.

Buck workflow gives an AI agent a project notebook, a filing system, and a checklist.
The agent can still reason and code — but important context is no longer trapped in the chat.

— The core design principle: don't lose work

Philosophy

Pick Up Where You Need To

Buck workflow is not a rigid pipeline you must run from start to finish. It's a toolkit of named thinking modes. Use the ones your task needs. Skip the rest. Start in the middle. Finish early. The artifacts and global conventions make it work regardless of which stages you run.

Full Workflow — New Feature

End-to-end for complex, multi-session work

brainstorm explore research plan build review save

A new feature from scratch. Brainstorm captures the vague idea, explore investigates the codebase, research checks external APIs, plan scopes the work, build implements, review validates, save records. Every artifact is durable.

Partial — Plan + Review

You already know what to build, you want a plan and a review

plan review save

Skip the exploration. Write a plan, build freely, then review against it. The plan creates the subject folder and the review writes iteration artifacts if issues are found. Save commits it all to memory.

Partial — Brainstorm + Plan + Build

Idea to implementation in one session

brainstorm plan build

Interview yourself, get a plan on paper, then build it. Skip review if the change is small enough. The brainstorm and plan still create a subject folder so the thinking survives even if you don't save.

Quick Fix Loop

Small change, verify, done

build review

Sometimes you just need to fix a bug. Build the smallest safe change, review it, commit. No plan, no research, no phases. The global AGENTS.md still ensures memory gets written after significant work.

How this works without running every stage

The global ~/.pi/agent/AGENTS.md ensures that even without any Buck commands, the agent still follows core conventions: check .context/ before starting, write memory after significant work, update the backlog. Buck workflow stages add structure on top. The global baseline means you always get durability even in quick ad-hoc sessions.

What's Pi-specific vs portable

Prompt templates (/b-* commands) are a Pi feature. Skills (the canonical workflow logic in skills/*/SKILL.md) are agent-neutral Markdown files — they work in Claude Code, Cursor, OpenCode, and Codex through each agent's native command mechanism. Extensions (b-flow, /b-save, write guards) are Pi-specific runtime automation. The global AGENTS.md and .context/ conventions are fully portable.

The Foundation

Global AGENTS.md — The Always-On Baseline

Before any Buck command runs, the global ~/.pi/agent/AGENTS.md establishes the conventions that make the whole system work. It's loaded for every session, in every project. It tells the agent: check context, write artifacts, don't lose work.

~/.pi/agent/AGENTS.md — the critical workflow section (abridged)
## ⚠️ CRITICAL WORKFLOW - READ FIRST

**Before starting ANY task, execute these steps:**

1. ✅ Check memory index, then read recent memory files (3-5 most recent)
2. ✅ Acknowledge context to the user

**After completing ANY significant work:**

3. ✅ Write a persistent artifact to .context/
4. ✅ Write session memory to .context/memory/<topic>-YYYY-MM-DD.md
5. ✅ Update memory index (.context/memory/index.md)
6. ✅ Update backlog status
7. ✅ Update completed specs/plans

Why this matters

The global AGENTS.md means that even without running any Buck command, the agent still creates durable artifacts. You can work freeform, run just /b-plan, or do a full workflow — the baseline ensures memory, backlog, and subject folders are always maintained. Buck stages add structure; the global baseline ensures continuity regardless.

Runtime Orchestration

How b-flow Tracks Progress Automatically

The b-flow extension runs an XState state machine that persists workflow state across sessions. It scans .context/, detects where you left off, and routes the next command to the right stage. You don't have to remember where you were — the machine does.

extensions/b-flow/machine.ts — actual XState setup
import { assign, createActor, setup } from "xstate";
import { hasGoal, requiresReview, requiresReplan } from "./guards.js";

function defaultContext(projectRoot: string) {
  const saved = readProjection(projectRoot); // persists across sessions
  if (saved) return { goal, subject, projection: saved, ... };
  return { projection: { currentState: "idle", history: [], queue: [] } };
}

Safety Mechanism

Write Boundaries Prevent Premature Action

Each stage has explicit write boundaries. This is the engineering control that stops the agent from editing source code while still researching, or skipping review after building. The boundaries are enforced both by skill instructions and by the /b-mode extension at runtime.

skills/b-plan/SKILL.md — write boundary rules
## Write Boundary

- You may write to `.context/**` and temporary scratch locations.
- Do not modify source files outside `.context/`.
- **Allowed**: Native write and edit tools for `.context/**` files.
- **Allowed**: Bash commands for `.context/**` directory operations.
- **Blocked**: Bash redirects and file modifications outside `.context/**`.
extensions/index.ts — runtime write guard
const PLAN_MODE_ALLOWED_PATHS = [".context/", "docs/"];

const SAFE_BASH_PATTERNS: RegExp[] = [
  /^\s*cat\b/, /^\s*ls\b/, /^\s*grep\b/, /^\s*find\b/,
  /^\s*head\b/, /^\s*tail\b/, /^\s*git\s+(status|log|diff)\b/,
];

Engineering detail — two layers of enforcement

Write boundaries are enforced at two levels. First, the skill instructions tell the agent what it may and may not write (soft guard — the LLM follows instructions). Second, the /b-mode extension intercepts tool calls at runtime and blocks writes to disallowed paths (hard guard — the extension rejects the call). This dual approach means even if the agent "forgets" the boundary, the extension catches it.

Benefits

Feature → Benefit → Why You Should Care

01

Continuity

Subject folders with index.md entrypoints create a stable workspace. Anyone — human or agent — can find the current story without rereading chat.

"Work survives chat compaction, lunch breaks, weekends, and agent handoffs."

02

Trust & Auditability

Memory files with frontmatter create an auditable timeline. Every decision, abandoned approach, and test result is on record.

"Evidence trail for decisions and outcomes."

03

Repeatability

Skills encode reusable guidance. Fix a workflow once, it stays fixed across projects and agents — Pi, Claude Code, Cursor, Codex.

"Improve the system once, benefit every future run."

04

Quality Control

Named stages with write boundaries prevent premature coding and skipped review. Findings become iterate-*.md artifacts, not vanishing comments.

"Quality feedback becomes part of the workflow state."

05

Lower Rediscovery Cost

Subject entrypoints and memory reduce repeated context gathering. Future sessions start faster because they can read what was already learned.

"Each session starts closer to the problem."

06

Handoff & Scaling

b-phase decomposes large plans into safe, resumable chunks. New engineers or agents pick up where the last session ended.

"Quick fix to multi-session — same workflow."
