The Problem

Why Chat-Only Agent Work Breaks Down

AI agents are fast — but speed is not the same as continuity, trust, or repeatability. When the conversation ends, the context evaporates.

🕳️

Lost Context

Chat windows compact. Sessions expire. Critical decisions vanish into scrollback. Next session starts from scratch.

👁️

No Audit Trail

"Why did we do this?" — the only answer lives in someone's chat history. No paper trail, no decision record, no accountability.

🔁

Repeated Discovery

Each new session re-explores the same codebase, re-learns the same constraints, re-makes the same mistakes.

⚡

Premature Coding

Without guardrails, agents jump to editing files before understanding the problem. Research, planning, and review get skipped.

🤝

No Handoff

Work can't be resumed by another engineer or agent. Everything is trapped in one conversation's context window.

📋

No Quality Gates

Review is optional and easily skipped. Feedback lives in chat comments, not as actionable follow-up items.

Buck workflow gives an AI agent a project notebook, a filing system, and a checklist.
The agent can still reason and code — but important context is no longer trapped in the chat.

Workflow Stages

Named Thinking Modes

Exploration, research, planning, building, reviewing, and saving are different kinds of thinking. Buck gives each one a named entrypoint and clear write boundaries — so the agent does the right kind of work at the right time.

💡 /b-brainstorm

Intake

Interview-style capture of vague ideas. No pretending they're ready.

→
🔍 /b-explore

Explore

Investigate the internal codebase. Map architecture. Read-only.

→
📚 /b-research

Research

External facts: APIs, docs, libraries. Capture evidence.

→
📐 /b-plan

Plan

Bounded intent: scope, files, risks, verification steps.

→
🔧 /b-build

Build

Implement with TDD. Red-green-refactor. Smallest safe change.

→
✅ /b-review

Review

Validate against plan. Write iteration artifacts if issues found.

→
💾 /b-save

Save

Convert session into durable history. Update memory and index.

🗂️ The Cross-Reference System

Every artifact carries forward and backward links in its frontmatter. A plan links to the research that informed it. A memory file links to the subject folder it relates to. An agent with zero conversation history can navigate from any artifact to the full picture.

Runtime Orchestration

How b-flow Tracks Progress

Behind the scenes, the b-flow extension runs an XState state machine that persists workflow state across sessions. It scans .context/, detects where you left off, and routes the next command to the right stage.

extensions/b-flow/machine.ts — XState state machine definition
import { assign, createActor, fromPromise, setup } from "xstate";

function defaultContext(projectRoot: string): BuckMachineContext {
  const saved = readProjection(projectRoot);
  if (saved) return { goal, subject, projection: saved, ... };
  return {
    goal: "",
    projection: {
      currentState: "idle",
      history: [],
      queue: [],
    },
  };
}

Benefits

Feature → Benefit → Why You Should Care

Every feature in Buck workflow maps to a concrete benefit. Here's the translation.

01

Continuity

Subject folders with index.md entrypoints create a stable workspace for an initiative. Anyone — human or agent — can find the current story without rereading a chat transcript.

"Work survives chat compaction, interruptions, and agent handoffs."

02

Trust & Auditability

Memory files with frontmatter create an auditable timeline. The team can answer "how did we get here?" — not just "where did we end up?" Plans, decisions, reviews, and outcomes are all on record.

"Evidence trail for decisions and outcomes — not just output."

03

Repeatability

Skills encode reusable guidance. Prompt templates expose commands. Extensions automate bookkeeping. Teams improve the system once and that improvement shows up in every future workflow run.

"Agent behavior becomes teachable and improvable."

04

Quality Control

Named stages with write boundaries prevent premature coding while researching, and prevent skipping review after implementation. Review findings become durable iteration artifacts, not vanishing chat comments.

"The workflow prevents premature coding, skipped review, and lost decisions."

05

Lower Rediscovery Cost

Subject entrypoints and memory reduce repeated context gathering. Future sessions start faster, waste fewer tokens, and are less likely to repeat old mistakes because they can read what was already learned.

"Each session starts closer to the problem — not from scratch."

06

Handoff & Scaling

b-phase decomposes large plans into independently verifiable phases. Work can proceed in smaller, safer, resumable chunks. New engineers (or new agents) can pick up from where the last session ended.

"The workflow scales from a quick fix to multi-session work without losing the thread."

Safety Mechanism

Write Boundaries Prevent Premature Action

Each workflow stage has explicit write boundaries. Exploration is read-only. Planning can only write to .context/. Building can modify source code. Review writes iteration artifacts if issues are found. This prevents the agent from jumping ahead.

skills/b-plan/SKILL.md — write boundary rules
## Write Boundary

- You may write to `.context/**` and temporary scratch locations.
- Save plans where the user can reuse them outside the context window.
- Do not modify source files outside `.context/`.
- **Allowed**: Native write and edit tools for `.context/**` files.
- **Allowed**: Bash commands for `.context/**` directory operations.
- **Blocked**: Bash redirects and file modifications outside `.context/**`.
extensions/index.ts — plan mode write guard configuration
const PLAN_MODE_ALLOWED_PATHS = [".context/", "docs/"];

const SAFE_BASH_PATTERNS: RegExp[] = [
  /^\s*cat\b/, /^\s*ls\b/, /^\s*grep\b/, /^\s*find\b/,
  /^\s*head\b/, /^\s*tail\b/, /^\s*git\s+(status|log|diff)\b/,
];

Why It Matters

What This Means For You

👷 For Engineers

  • The workflow teaches good habits: investigate first, plan explicitly, build narrowly, review, then record what happened
  • Work survives your lunch break, your weekend, or your context window hitting its limit
  • Another engineer (or another agent) can pick up exactly where you left off
  • Skills are improvable — fix a workflow once and it stays fixed across projects
  • TDD is built into /b-build with red-green-refactor loops

📊 For Stakeholders

  • You get faster AI-assisted delivery without giving up traceability or reviewability
  • Every decision has a paper trail — plans, research, reviews, and memory are all durable files
  • /b-present turns the engineering record into an async-readable briefing
  • Risk is reduced: scope is visible before implementation, review is mandatory before commit
  • The system scales from a quick fix to a multi-week initiative with the same workflow
