# OMP-Workflow

$ARGUMENTS

Drop the keyword workflow in your next prompt. omp will inject a notice
that switches you into the eval-kernel fan-out contract.

In the persistent Python cell, use the prelude helpers:

- `agent(prompt, *, agent_type, schema)` — one subagent; with `schema=` returns parsed JSON
- `parallel(thunks)` — bounded pool (width = `task.maxConcurrency`)
- `pipeline(items, *stages)` — barrier between stages
- `llm(prompt, *, model, schema)` — oneshot judge; tiers: smol / default / slow
- `budget` — `.total`, `.spent()`, `.remaining()`; respect the ceiling

Patterns: adversarial verify, perspective-diverse verify, judge panel,
loop-until-dry, multi-modal sweep, completeness critic.

See `.context/<subject>/eval-<topic>.py` if a starter cell was generated by b-plan.

# Starter eval cell — generated by b-plan when omp_execution: workflow
# User edits this before invoking the `workflow` keyword.

from prelude import agent, parallel, pipeline, llm, phase, log, budget

phase("fan-out: N finders, each searching a different dimension")

def finder(dimension):
    return agent(
        f"Find issues in {dimension}. Return JSON: {findings: [...], severity: ...}",
        agent_type="task",
        schema={"type": "object", "properties": {"findings": {"type": "array"}}},
    )

dimensions = ["correctness", "regressions", "performance", "security"]
raw = parallel([lambda d=d: finder(d) for d in dimensions])

phase("synthesize: judge panel — keep findings that survive majority")

def judge(finding):
    verdict = llm(
        f"Is this a real issue? {finding}. Reply yes/no with one-sentence reason.",
        model="smol",
    )
    return {"finding": finding, "verdict": verdict}

kept = pipeline(raw, lambda r: [judge(f) for f in r["findings"]])

log(f"audit complete: {len(kept)} findings kept; budget remaining {budget.remaining()}")

◆ Key Decisions

  • Integration is prompt-level / skill-level, not extension-based
    b-flow was deprecated 2026-06-01 for being wired-but-never-invoked. Lesson: don't ship runtime machinery the user has to discover. All five integration surfaces are SKILL.md and PROMPT.md changes the user runs from the TUI.
  • Default omp_execution: none (omitted field)
    Existing plans and ad-hoc work must keep working. The new field is opt-in; b-plan recommends a value, never auto-sets it; the user confirms.
  • b-build does NOT auto-insert the magic keyword
    omp guards if (!options?.synthetic) at agent-session.ts:4274. Synthetic / agent-initiated turns never trigger the magic-keyword notices. The user must say the keyword. The phase file is a recommendation, not a directive.
  • Python (not JavaScript) for the eval cell template
    omp's workflow-notice examples are in Python; the prelude's agent()/parallel()/pipeline() are in src/eval/py/prelude.py. Python is the documented surface. JS is a follow-up if asked.
  • Cross-reference the b-flow deprecation in every new doc/skill
    The next agent who reads F4-F5 must see "we already tried extension-based orchestration; it didn't work; here's why we're not doing that again." Otherwise the same mistake re-emerges.
  • Goal-mode state is session-scoped, not plan-scoped
    /goal set is a user-toggled runtime state. The plan can recommend a budget; the user must enter goal mode. A plan with 5 phases and omp_execution: goal means the user sets a single budget for the whole plan — phases compete for the same ceiling.

⚠ Risks & Tradeoffs

High
Auto-keyword insertion would silently no-op
Document the synthetic-guard constraint in every skill that touches omp. Phase files phrase "the user should drop the keyword" — never "the agent runs the keyword."
High
Repeating the b-flow mistake (extension that isn't invoked)
F1 stubs are independently discoverable from the TUI. F4-F6 are additions to the existing b-plan/b-phase/b-build surface that already runs. No new Pi extension, no new state machine.
Medium
Per-turn budget vs plan-wide budget confusion
b-plan output for omp_execution: goal plans explicitly states "single budget for the whole plan; phases share it." Goal mode's token_budget is goal-scoped, not phase-scoped.
Medium
Eval cell is interactive-TUI-only
workflow keyword is meaningless under --mode rpc and omp -p. b-plan surfaces this in the plan's "Ralph Instructions" section: "this mode requires the interactive TUI."
Medium
Orchestrate's "no-yield-between-phases" vs buck's per-session phase
Phase file's ralph_complexity: multi already encodes "phase may take multiple iterations." The new omp_execution: orchestrate field is per-phase, not plan-wide — each phase is a unit of "do not yield until phase done."
Low
Backward compatibility for existing plans
All new fields are optional with documented defaults. Existing phase files without omp_execution behave as none. Existing skills behave unchanged.
